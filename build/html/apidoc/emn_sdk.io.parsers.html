A block for data, which differs from key/value blocks.

Expected format:

row 1: ‘Data[:<name>]’, <other columns ignored> row 2: ‘<unit>’, ‘<unit>’, .. , <blank> row 3..N: value , value , ..

The number of columns in the block is determined by how many non-empty “units” columns there are under the header (first) row.

class emn_sdk.io.parsers.matrix_blocks.KeyValueBlock(row, i, j)[source]¶

Bases: emn_sdk.io.parsers.matrix_blocks.Block

A block for metadata, as key/value pairs.

Expected format:

row 1: ‘<Type>[:<subtype>]’, <ignored> row 2..N: <key>, <value>

The “:subtype” is optional, and may be any alphanumeric sequence (with dash or underscores). The “type” must match a known metadata category. The value of <subtype> will be pre-pended to all the <key>s in the block.

class emn_sdk.io.parsers.matrix_blocks.SearchDataBlock(row, i, j)[source]¶

Bases: emn_sdk.io.parsers.matrix_blocks.KeyValueBlock

SearchData blocks are like metadata Key/Value blocks, except the subtype is not automatically prepended to keys, and instead just available as a separate attribute.

Also, there is no merge_data() method, because you can only have one SearchData block for a given subtype (“SearchData:Alligator”, “SearchData:Bear”, are both ok, but you can’t have two “SearchData:Alligator” sections in the same input matrix).
